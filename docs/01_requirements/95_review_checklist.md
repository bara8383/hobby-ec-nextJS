# 95_review_checklist

本チェックリストは、要件定義の抜け漏れ・矛盾を検出するためのレビュー観点をまとめる。
設計・実装方式（技術名）には踏み込まない。

---

## 1. スコープ・フェーズ
- [ ] Phase1（MVP）とPhase2（将来）が明確に分離されている
- [ ] Phase1が「CtoC＋デジタルDL」で一貫している
- [ ] Phase2（物販等）は要件に混入していない

## 2. ロールと責任分界
- [ ] 購入者/出品者/運営者の責務が明確
- [ ] 返金は運営が一元対応であることが明文化されている
- [ ] 審査・通報・停止の最終判断者が運営であることが明確

## 3. 商品（出品）要件
- [ ] 商品必須属性（商品名/短文説明/価格/提供ファイル/販売状態/注意事項）が要件に記載されている
- [ ] 短文説明120文字上限・超過時自動切り詰め（暫定）が記載されている
- [ ] 初回出品のみ審査が要件として定義されている
- [ ] 販売停止後60日DL可→提供終了が明文化されている

## 4. 購入・注文・状態遷移
- [ ] 複数商品まとめ買い（カート方式）が要件に含まれている
- [ ] 注文状態（購入手続き中/支払い待ち/購入成立/キャンセル/返金済み）が定義されている
- [ ] 状態遷移の整合（支払い待ち→期限切れ→キャンセル 等）が破綻していない
- [ ] 支払い待ち期限の起算点（支払い待ちになった時点）が明確
- [ ] 支払い待ち期限は区分別に設定可能である
- [ ] 期限日時表示が要件に入っている
- [ ] 期限切れ後の支払いは無効（キャンセルのまま）が要件に入っている

## 5. 提供（DL）とDLライブラリ
- [ ] DLライブラリは購入成立のみ表示である
- [ ] 提供期限なし（永久DL可）と再DL無制限が要件化されている
- [ ] DL済み判定が「提供手続き初回実行」である
- [ ] 退会後はDL不可である（アクセス停止）が要件に整合している

## 6. 返金
- [ ] 未DLのみ返金可、DL済みは通常返金不可が要件にある
- [ ] 返金例外（不正/悪用/規約違反）は個別対応（裁量あり）が要件にある
- [ ] 返金完了通知が要件にある
- [ ] 返金済みはDLライブラリに表示しないことと矛盾していない
- [ ] （CtoC特有）返金コストは運営負担であることが明文化されている

## 7. 通知（チャネル・サイト内通知）
- [ ] チャネルはメール＋サイト内通知である
- [ ] チャネルは選択式だが最低1つONの制約がある
- [ ] デフォルト両方ONが要件にある
- [ ] サイト内通知：一覧＋既読/未読が要件にある
- [ ] サイト内通知：ユーザー削除可が要件にある
- [ ] サイト内通知：既読のみ90日で自動削除、未読は自動削除しない が要件にある
- [ ] 通知一覧：ページングあり、1ページ50件が要件にある
- [ ] 販売停止通知（即時＋7日前、全購入者）が要件にある
- [ ] 返金完了通知が要件にある

## 8. 通報・措置（CtoC）
- [ ] 通報理由（権利侵害/禁止商材/詐欺・虚偽説明）が要件にある
- [ ] 運営の措置（クローズ/差し戻し/一時停止/削除/出品者停止）が要件にある
- [ ] 通報者へのフィードバック：受付完了＋最終結果＋簡易理由が要件にある
- [ ] 措置後の表示（閲覧不可/購入不可等）の基本方針が矛盾していない

## 9. 手数料・精算（CtoC）
- [ ] 手数料モデル：販売額の割合（%）が要件にある
- [ ] 一律適用（全出品者）が要件にある
- [ ] 手数料率：暫定10% が要件にある
- [ ] 手数料率変更は新規注文のみ適用（遡及しない）が要件にある
- [ ] 精算は週次である
- [ ] 日曜締め→水曜払いが要件にある
- [ ] 出品者への精算明細は注文単位である

## 10. アカウント（会員）
- [ ] 会員登録必須（ゲスト購入なし）が要件にある
- [ ] 登録必須項目（メール/パスワード/規約同意/メール認証）が要件にある
- [ ] パスワード再設定が要件にあり、有効期限1時間が要件にある
- [ ] メール変更は認証完了まで無効が要件にある
- [ ] 退会が要件にあり、退会後は注文履歴閲覧不可が要件にある
- [ ] プロフィール項目を持たない方針が要件にある

## 11. エラー/例外（基本方針）
- [ ] 主要例外（期限切れ、DL不可、返金不可、審査差し戻し、通報失敗等）が網羅されている
- [ ] 結果不明時は安全側（重複実行回避）になっている
- [ ] 問い合わせ導線が示される前提になっている

## 12. ドキュメント整合・表現
- [ ] 同じ概念が複数の用語で揺れていない（例：DL済み判定/提供手続き等）
- [ ] 要件に技術名や実装方式が混入していない
- [ ] 未決事項は risks（99_open_issues）に隔離されている
- [ ] 画面一覧・ユースケース一覧・業務ルール・機能要件が相互に矛盾しない
